# OPC Flow Quickstart

> 让 `CEO -> COO -> sub-agent` 不只是角色表，而是一套可复用的推进方式。

## 三种模式

| 模式 | 适用场景 | 核心角色 | 目标时长 |
|---|---|---|---|
| `OPC-Micro` | 单个问题、单次决策、小范围验证 | CEO / COO / 1-3 个 sub-agent | 0.5-1 天 |
| `OPC-Sprint` | MVP、功能迭代、业务专项 | CEO / COO / 8 个核心 sub-agent | 1-3 周 |
| `OPC-Control` | 多团队、跨部门、持续运行场景 | 满编 20 角色协同 | 持续 |

默认编组建议：

- `OPC-Micro` 对应 `daily`，常驻 3 个 sub-agent
- `OPC-Sprint` 对应 `important`，拉起 8 个核心 sub-agent
- `OPC-Control` 对应 `full`，直接启用 20 角色满编协同

## OPC-Micro

适用于单个问题快速收敛：

1. CEO 收到问题，做目标拆解
2. COO 定级并确定状态机
3. 策略/研究/技术等角色并行补充事实和方案
4. QA 或法务做边界校验
5. CEO 汇总主结论

推荐提示：

```text
以 OPC-Micro 模式运行这个任务。
先由 CEO 主控拆解，再由 COO 定级和推进。
需要时派发给最少数量的 sub-agent，最后汇总主结论、风险和下一步动作。
```

## OPC-Sprint

适用于功能、MVP 和专项交付：

1. CEO 识别目标、边界、主要风险
2. COO 定级并创建状态机
3. 项目角色拆里程碑，产品/研究/体验补齐需求
4. 技术/运维/采购/财务补实现和资源约束
5. QA 做交付验收
6. 客户成功/数据分析做上线后验证

推荐提示：

```text
以 OPC-Sprint 模式推进这个需求。
必须包含：里程碑拆解、风险记录、阶段状态、验收口径。
所有角色输出最后由 CEO 汇总成主结论。
```

## OPC-Control

适用于持续运行或多部门联动：

1. CEO 负责阶段目标和升级判断
2. COO 负责状态、节奏、同步、复盘
3. 项目角色维护里程碑和阻塞
4. 财务、法务、采购、HR、客户成功持续补边界
5. 数据分析回填指标，记忆同步沉淀经验

推荐提示：

```text
以 OPC-Control 模式接管这个事项。
要求主从编排、持续状态同步、风险量化、记忆回填。
任何阻塞或红线问题都要升级给 CEO 主 agent。
```

## 核心原则

- 没有定级，不进入多人协作
- 没有派发上下文，不允许 sub-agent 冷启动
- 没有证据或验收口径，不算完成
- 没有风险和下一步动作，不允许 CEO 直接收尾
